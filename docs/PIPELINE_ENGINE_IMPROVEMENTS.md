# Pipeline & Engine Improvement Roadmap

**Date:** 2026-04-05
**Scope:** GameDev monorepo pipeline (Python tools → GLB) and VibeGame engine integration.

---

## Executive Summary

The monorepo has a solid pipeline: Python tools generate assets (GLB, textures, audio) and VibeGame loads them in the browser. However, there are **gaps** between generation and runtime that cause friction. This document identifies concrete improvements organized by impact.

---

## 1. Pipeline → Engine Integration Gaps

### 1.1 🔴 No Shared Asset Manifest Format

**Problem:** GameAssets outputs files, but VibeGame has no standard way to discover what's available at runtime. The `handoff` step copies files, but there's no machine-readable manifest that the engine can consume.

**Proposal:** Define a `gameassets_manifest.json` that the engine reads:

```json
{
  "version": 1,
  "generated": "2026-04-05T20:00:00Z",
  "assets": {
    "hero": {
      "model": "assets/models/hero.glb",
      "textures": ["assets/textures/hero_basecolor.png"],
      "animations": ["BreatheIdle", "Walk", "Run"],
      "audio": "assets/audio/hero_voice.wav"
    }
  }
}
```

**Impact:** High — enables dynamic loading, scene composition from YAML/CSV, and `gameassets dream` output → engine input without manual wiring.

**Effort:** ~2-3 days

### 1.2 🔴 Missing GLB Metadata in Pipeline

**Problem:** When Text3D/Rigging3D/Animator3D produce GLBs, the engine doesn't know:
- Which clips are embedded (for animation UI)
- What the model bounds are (for camera placement)
- PBR texture set associations

**Proposal:** Add a `glb-metadata.json` sidecar to each GLB output with:
```json
{
  "file": "hero.glb",
  "animations": ["BreatheIdle", "Walk"],
  "bounds": {"min": [-1, 0, -1], "max": [1, 2, 1]},
  "pbr_textures": ["hero_basecolor.png", "hero_normal.png"],
  "source_pipeline": "text3d → paint3d → rigging3d → animator3d"
}
```

**Impact:** High — enables smart camera framing, animation selection UI, and automatic PBR setup.

**Effort:** ~1-2 days (post-processing step after each tool)

### 1.3 🟡 Text2D Output Not Wired to VibeGame

**Problem:** Text2D generates sprites/textures but there's no VibeGame recipe for 2D sprites (billboards, UI elements, decals).

**Proposal:** Add a `<sprite>` recipe to VibeGame:
```xml
<sprite texture="/assets/textures/hero_portrait.png" width="2" height="2" billboard="true" />
```

**Impact:** Medium — unlocks 2D UI elements generated by AI.

**Effort:** ~1 day

### 1.4 🟡 Skymap2D → VibeGame Skybox Bridge

**Problem:** `applyEquirectSkyEnvironment` exists but requires manual wiring. No automatic discovery from handoff output.

**Proposal:** In the handoff step, auto-generate a `<skybox>` XML snippet or export a scene config that includes the sky:
```xml
<skybox url="/assets/sky/fantasy_forest.png" blur="0.5" />
```

**Impact:** Medium — one-click from generation to rendered sky.

**Effort:** ~0.5 day

### 1.5 🟡 Audio (Text2Sound) Not Integrated

**Problem:** Text2Sound generates WAV files but VibeGame has no audio playback system.

**Proposal:** Add an `AudioPlugin` with spatial audio:
```typescript
state.registerPlugin(AudioPlugin);
const entity = state.createEntity();
state.addComponent(entity, AudioSource, { url: "/assets/audio/ambient.wav", loop: true, volume: 0.5 });
```

**Impact:** Medium — completes the full pipeline (visuals + audio).

**Effort:** ~2 days

## 2. CI/CD Pipeline Improvements

### 2.1 🔴 CI Takes Too Long (~5-7 min)

**Problem:** 18 parallel pytest jobs, each installing Shared + package from scratch.

**Proposal:** Use `actions/cache` for pip packages and Python builds:
```yaml
- uses: actions/cache@v4
  with:
    path: |
      ~/.cache/pip
      ~/.local/lib/python*/site-packages
    key: pip-${{ matrix.python-version }}-${{ hashFiles('Shared/pyproject.toml', '*/pyproject.toml') }}
```

**Impact:** High — could cut CI from ~5 min to ~2 min.

**Effort:** ~1 hour

### 2.2 🟡 No CI Caching for Bun/VibeGame

**Problem:** Every VibeGame CI run does `bun install --frozen-lockfile` from scratch.

**Proposal:** Add bun cache:
```yaml
- uses: actions/cache@v4
  with:
    path: ~/.bun/install/cache
    key: bun-${{ runner.os }}-${{ hashFiles('VibeGame/bun.lock') }}
```

**Impact:** Medium — saves ~30s per run.

**Effort:** ~30 min

### 2.3 🟡 No Release / Version Automation

**Problem:** Versions are manually bumped in 15+ `pyproject.toml` files + `package.json`.

**Proposal:** Add a `release` workflow that:
1. Bumps versions via `standard-version` or custom script
2. Creates git tags
3. Publishes to PyPI (Python) and npm (VibeGame)

**Impact:** Medium — enables proper releases and changelog.

**Effort:** ~2 days

### 2.4 🟢 Missing E2E Test

**Problem:** CI tests each package in isolation. No test that validates the full pipeline (Text2D → Text3D → GLB → VibeGame load).

**Proposal:** Add a minimal E2E test that:
1. Generates a tiny mesh (Text3D mock/synthetic)
2. Validates GLB structure with `trimesh`
3. Loads it in VibeGame via `loadGltfToScene` in jsdom

**Impact:** Low-Medium — catches integration breakage.

**Effort:** ~2 days

### 2.5 🟢 Dependabot Not Configured for Python

**Problem:** Only `actions/checkout` and Cargo deps have Dependabot. Python packages and Bun deps are not tracked.

**Proposal:** Add to `.github/dependabot.yml`:
```yaml
- package-ecosystem: "pip"
  directory: "/Shared"
- package-ecosystem: "npm"
  directory: "/VibeGame"
```

**Impact:** Low — automated security updates.

**Effort:** ~15 min

## 3. VibeGame Engine Improvements

### 3.1 🔴 No Scene Composition from Manifest

**Problem:** Loading assets into VibeGame requires manual TypeScript code for each entity. No way to compose a scene from a data file.

**Proposal:** Add `<scene-include>` that loads a JSON manifest:
```xml
<scene-include src="/assets/gameassets_manifest.json" />
```
Or a programmatic API:
```typescript
await loadManifest(state, '/assets/gameassets_manifest.json');
```

**Impact:** High — turns the pipeline into a "generate → play" workflow.

**Effort:** ~3 days

### 3.2 🟡 No LOD (Level of Detail) System

**Problem:** Models loaded from the pipeline have no LOD variants. Browser performance degrades with many high-poly models.

**Proposal:** Add a `<lod>` component:
```xml
<lod near="0" far="50" near-model="/assets/models/hero_hires.glb" far-model="/assets/models/hero_lod.glb" />
```

**Impact:** Medium — critical for games with many 3D objects.

**Effort:** ~2 days

### 3.3 🟡 No Asset Loading Progress / Streaming

**Problem:** Large GLBs from Text3D can be 10-50MB. No loading indication in the engine.

**Proposal:** Add loading progress callbacks to `loadGltfToScene`:
```typescript
const { group, progress } = await loadGltfToScene(state, url, {
  onProgress: (loaded, total) => console.log(`${(loaded/total*100).toFixed(0)}%`),
});
```

**Impact:** Medium — better UX for web games.

**Effort:** ~0.5 day

### 3.4 🟡 No Physics Integration from Pipeline

**Problem:** Part3D decomposes meshes into parts, but there's no way to use those as physics colliders in VibeGame.

**Proposal:** Add a bridge from Part3D output → VibeGame physics colliders:
```xml
<collider type="convex-hull" src="/assets/models/hero.physics.glb" />
```

**Impact:** Medium — enables collision detection from generated assets.

**Effort:** ~2 days

### 3.5 🟢 No Texture Hot-Reload

**Problem:** Re-running Texture2D requires restarting the VibeGame dev server.

**Proposal:** Add Vite plugin that watches `public/assets/textures/` and reloads materials:
```typescript
// vibegame.config.ts
export default {
  hotReload: {
    textures: true,
    models: false, // GLBs are too large for hot reload
  }
};
```

**Impact:** Low — faster iteration during development.

**Effort:** ~1 day

## 4. Python Pipeline Improvements

### 4.1 🟡 No Pipeline Validation Step

**Problem:** Each tool (Text2D, Text3D, Paint3D, etc.) validates its own output, but there's no cross-tool validation. A broken Text3D output can crash Paint3D downstream.

**Proposal:** Add a `gamedev validate` command that checks:
- GLB is valid (trimesh load + basic mesh checks)
- Required textures exist and are valid images
- Animation clips have valid duration > 0
- Total file size is reasonable

**Impact:** Medium — catches errors early in the pipeline.

**Effort:** ~2 days

### 4.2 🟡 No Pipeline Caching

**Problem:** Re-running `gameassets batch` regenerates everything from scratch. If only one asset changes, all are regenerated.

**Proposal:** Add content-addressed caching:
- Hash input params + source data → cache key
- Skip regeneration if cache hit
- `--force` flag to bypass cache

**Impact:** Medium — huge time savings for iterative asset work.

**Effort:** ~3 days

### 4.3 🟢 Version Alignment

**Problem:** `gamedev-shared` is 0.2.0 but packages don't declare minimum shared version.

**Proposal:** Add `gamedev-shared>=0.2.0` to each package's dependencies instead of just `gamedev-shared`.

**Impact:** Low — prevents version mismatch bugs.

**Effort:** ~30 min
